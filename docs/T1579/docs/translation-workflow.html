Translation Workflow

白話翻譯工作流程

本頁記錄後續卷次翻譯時的 agent 分工、術語保存方式、交付格式與檢查流程。

術語庫 JSON 翻譯清單 來源稿範例

目標

每一卷都以 CBETA 原文為底本，產出可追溯的 Markdown 來源稿與左右對照 HTML。翻譯以正確精準優先，通順漂亮其次；核心術語以玄奘譯語為準，必要處使用「白話詞（玄奘詞）」雙軌。

多 agent 的目的不是讓每個人各自改主檔，而是把切段、初譯、術語、義理、外部對照、格式檢查分開，最後由 coordinator 統一合併。

視覺流程

1. 鎖範圍 Coordinator 確定卷號、CBETA 起訖、卷界與產出檔名。
2. 切段 Segmenter 先寫 segment TSV，再由工具抽 Range、Source 與邊界 note。
3. 分段初譯 Translators 只填指定段落的白話與必要校註。
4. 三路審核 術語、義理、外部對照各自輸出建議，不直接覆蓋主檔。
5. 合併發佈 Coordinator 定稿，HTML QA 建置、檢查、commit、push。

術語線

比對 glossary，處理同詞多譯、禁譯、首次雙軌顯示與跨卷延續。

義理線

檢查修道次第與教義關係，例如伏斷、永害種子、近分、根本定。

外部對照線

查梵文、巴利、英文或平行經論，只提交證據包與疑義判斷。

固定檔案

檔案 用途
translations/segments/T1579-XXX.tsv 每卷段落切分表，保存段題、CBETA range、邊界 note，供來源稿骨架重建。
translations/T1579-XXX-baihua.md 每卷唯一來源稿，保存段落 ID、CBETA range、原文、白話、校註。
html/translations/T1579-XXX-baihua.html 由來源稿產生的左右對照閱讀頁。
translations/glossary/T1579-terms.json 跨卷術語庫，保存玄奘詞、白話譯法、外語候補、語境規則與禁譯提醒。
scripts/make_translation_skeleton.py 依 CBETA JSON 與 segment TSV 產生 Markdown 來源稿骨架。
scripts/build_translation_html.py 將翻譯來源稿轉成 HTML。
scripts/check_translation_terms.py 檢查術語庫 schema 與來源稿中的基本術語覆蓋。
scripts/check_translation_coverage.py 檢查來源稿 range 單調性、起訖行號與 CBETA data line id 是否一致。
scripts/check_html_links.py 檢查 html/ 下所有相對 href/src 連結是否指向存在的檔案。

來源稿合約

所有 agent 都要遵守下列 Markdown 格式。Range、Source、Translation 必填，Note 可空但建議保留。下方以卷33為例。新卷先寫 translations/segments/T1579-XXX.tsv，再用 make_translation_skeleton.py 抽出 Source，不要手動複製 CBETA 原文。

## 01 段落標題
Range: T30n1579_p0465a23-p0465a27

Source:
<<<
CBETA 原文
>>>

Translation:
<<<
白話譯文
>>>

Note:
<<<
校勘、段落歸屬或譯語判斷。
>>>

Range 和 Source 只由切段 agent 或 coordinator 修改。翻譯 agent 若發現範圍問題，只在交接清單提出，不直接改邊界。

建立新卷骨架時使用下列指令。骨架工具預設拒絕覆蓋已存在的來源稿；真的要重建定稿才加 --force。

python3 scripts/make_translation_skeleton.py --juan XXX --data data/T1579-XXX.json --segments translations/segments/T1579-XXX.tsv --output translations/T1579-XXX-baihua.md --start T30n1579_p... --end T30n1579_p...

Agent Pool

Coordinator

鎖定卷號、底本範圍、段落分派與最後合併。所有 reviewer 的建議由 coordinator 收斂。

Segmenter

依 CBETA JSON、目次與語義切段，產出空白來源稿骨架。跨行、跨目次、卷首卷末都要寫入 note。

Translator

只填指定段落的 Translation 與必要 Note。不改 Range、不改別人的段。

Terminology Reviewer

檢查核心術語一致性，更新或建議更新 glossary。特別注意作意、勝解、補特伽羅、止觀、加行、麁性、靜性、伏斷。

Parallel Source Reviewer

查可公開取得的梵文、巴利、英文、Nikaya 或 Agama 平行資料，只交付證據包與疑義，不直接重寫主譯。

Doctrinal Reviewer

檢查義理與修道次第，例如世間道和出世間道、伏斷和永害種子、無想定和滅盡定是否混淆。

Coverage QA

比對行號範圍，確認沒有漏譯、重複、混入下一卷。段落邊界問題回報 coordinator。

HTML QA

跑 build、連結檢查、術語檢查與手機版結構檢查，確認生成頁可讀。

預設雙稿模式

每個正式翻譯任務預設採用雙稿模式：Codex 與 GLM 先各自產出白話初稿、疑義清單與術語判斷，Coordinator 只做格式整理與差異對照，不先融合立場。

Claude 擔任 final reviewer，負責比較 Codex/GLM 雙稿、CBETA 原文、glossary 與 reviewer 證據包，最後輸出定稿建議、保留疑義與必要 glossary 更新。

若 Claude 不可用，改由 Codex 執行 final review。此時 Codex 必須以 final reviewer 身分重新審讀雙稿，明確列出採用、改寫或退回 GLM 意見的理由，避免只沿用自己的初稿。

工作順序

  1. Coordinator 鎖定卷號、CBETA 起訖行號、是否保留卷題與卷末題。
  2. Segmenter 產生 translations/segments/T1579-XXX.tsv，所有段落都有標題、Range 和邊界 note。
  3. Coordinator 執行 scripts/make_translation_skeleton.py 產生 Markdown 骨架，所有段落都有 Range 和 Source。
  4. Coordinator 分派段號給 translators；每個 translator 只負責自己的段。
  5. Translator 依 glossary 初譯，預設由 Codex 與 GLM 各自提交雙稿；疑難處短註，不寫長篇論文式考證。
  6. Terminology Reviewer 檢查同詞多譯與禁譯，提出 glossary 更新。
  7. Parallel Source Reviewer 交付外部資料證據包，標明可信度與影響句。
  8. Doctrinal Reviewer 檢查義理、修道位階與上下文。
  9. Claude 執行 final review；若 Claude 不可用，改由 Codex 以 final reviewer 身分審讀雙稿並輸出定稿建議。
  10. Coordinator 統一合併、定稿、更新 glossary。
  11. HTML QA 跑建置與檢查，通過後 commit；有 remote 才 push。

術語庫規格

術語庫使用 JSON，避免新增 Python 依賴。每個條目至少要有下列欄位：

欄位 說明
id穩定機器 ID。
xuanzang玄奘原詞。
display建議首次或疑難處顯示形式。
plain主要白話譯法。
english、sanskrit、pali外語候補，只作對照，不取代 CBETA 底本。
source_terms在原文中要觸發檢查的詞。
translation_patterns譯文中可接受的譯法。
statusstable、review 或 deprecated。
rule語境規則、禁譯提醒或跨卷注意。

重要警告要寫進 rule。例如「勝解」不可只譯為一般理解，「伏斷」不可譯成究竟斷惑，「麁相／靜相」是上下地比較詞，不是固定實體性質。

必要檢查

python3 scripts/build_shengwen_di_html.py
python3 scripts/build_translation_html.py --translation translations/T1579-XXX-baihua.md
python3 scripts/check_translation_terms.py --translation translations/T1579-XXX-baihua.md
python3 scripts/check_translation_coverage.py --translation translations/T1579-XXX-baihua.md --data data/T1579-XXX.json --start T30n1579_p... --end T30n1579_p...
python3 scripts/check_html_links.py

check_html_links.py 會走訪 html/ 下所有頁面，確認 index.html、sections/、translations/、docs/ 沒有缺檔連結。

發布方式

公開網站目前使用 GitHub Pages：https://davidshih.github.io/Yogcarabhumi-sastra/。

.github/workflows/deploy-pages.yml 會在每次 push 到 main 時，把 html/ 目錄部署到 GitHub Pages。若需要手動重跑，可到 GitHub Actions 執行 Deploy Pages workflow。

若改用 S3，可同步 html/ 目錄作為靜態網站內容；但 HTTPS、自訂網域與快取通常還需要 CloudFront 與 DNS 設定。除非需要自訂 AWS 架構，預設維持 GitHub Pages。

衝突規則

  • 段落邊界只能由 Segmenter 或 Coordinator 改。
  • Reviewer 不直接覆蓋主檔，只提供建議、證據或 patch。
  • 術語修改要更新 glossary，不能只在單段譯文中局部改。
  • 外部英文、梵文、巴利資料只用來解疑，不作主底本。
  • 雙稿模式下，final reviewer 要保留採納理由；Claude 不可用時由 Codex 終審，不跳過終審步驟。
  • 測試通過後才 commit；沒有 remote 或 push 權限時，停在 commit 並回報。
